---
title: 'Self-hosting'
description: 'Run Floppa API directly on a server without Docker'
---

You can run Floppa API as a native binary on any Linux or macOS server. This approach gives you full control over the process and integrates cleanly with standard init systems like systemd.

## Prerequisites

- Go 1.24 or later
- [Bun](https://bun.sh/) (for building the frontend)
- A running [PocketBase](https://pocketbase.io/) instance

## Build and run

<Steps>
  <Step title="Build the frontend">
    From the repository root, install dependencies and build the Svelte frontend:

    ```bash
    cd frontend
    bun install
    bun run build
    ```

    This outputs static files to `frontend/dist/` (relative to the repo root).
  </Step>

  <Step title="Build the backend">
    Compile the Go binary from the `backend/` directory:

    ```bash
    cd backend
    go build -o floppa-api .
    ```

    This produces a `floppa-api` binary inside `backend/`.
  </Step>

  <Step title="Create config.json">
    Create `config.json` in the `backend/` directory (where the binary lives). The binary searches for the config file at `/config.json`, `config.json`, and `./config.json` in that order.

    ```bash
    cp backend/config.json.example backend/config.json
    ```

    Edit `backend/config.json` with your PocketBase connection details:

    ```json
    {
      "pocketbase_url": "http://your-pocketbase-host:8090",
      "pocketbase_admin": "admin@example.com",
      "pocketbase_password": "your_password"
    }
    ```
  </Step>

  <Step title="Run the binary">
    Run the binary from the **repository root** so that the relative path `./frontend/dist` resolves correctly:

    ```bash
    GIN_MODE=release ./backend/floppa-api
    ```

    The API listens on port `8080`. The frontend is served from `./frontend/dist`.

    <Note>
      The binary serves `./frontend/dist` relative to the directory it is run from. Always run from the repository root, or place the binary alongside a `frontend/dist/` directory in your deployment directory.
    </Note>
  </Step>
</Steps>

## Production setup with systemd

For a persistent production deployment, create a systemd service unit. This example assumes:
- The repository (or a copy of it) is deployed to `/opt/floppa-api`
- The compiled binary is at `/opt/floppa-api/backend/floppa-api`
- The built frontend is at `/opt/floppa-api/frontend/dist/`

Create the file `/etc/systemd/system/floppa-api.service`:

```ini
[Unit]
Description=Floppa API
After=network.target

[Service]
Type=simple
WorkingDirectory=/opt/floppa-api
ExecStart=/opt/floppa-api/backend/floppa-api
Environment=GIN_MODE=release
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

Then enable and start the service:

```bash
sudo systemctl daemon-reload
sudo systemctl enable floppa-api
sudo systemctl start floppa-api
```

Check the service status:

```bash
sudo systemctl status floppa-api
```

<Note>
  Set `WorkingDirectory` to the directory that contains the `frontend/dist/` folder. The binary serves `./frontend/dist` relative to the working directory — not relative to the binary itself.
</Note>

## Environment variables

| Variable   | Value     | Description                                    |
| ---------- | --------- | ---------------------------------------------- |
| `GIN_MODE` | `release` | Disables debug logging. Recommended for production. |
